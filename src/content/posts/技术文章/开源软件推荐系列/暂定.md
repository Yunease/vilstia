# 概念讲解

目前的主流ai部分已经实装了记忆功能，但是实现方式比较简陋，只是简单的增加了一个memory.md，并对这个文件不断读写。这会产生出几个显著的问题：

- 随着时间变长，这个文件的大小会不停增加，这会导致持续使用的token消耗量持续走高，不利于长期使用。
- 各种记忆存放的非常混乱，短期记忆和长期记忆无法区分，也没有访问频率和权重，极容易出现记忆混乱。
- 阅读检索效率较低。

我的这个项目就是为了解决这个问题而设计的，它分为如下层级：

- 用户UI层：现代化，清晰简洁的ui，支持自定义样式和持久化存储。
- agent架构层：对对话进行分析，提取记忆，并赋字段。（这意味着需要装几个简单的skills，也可以手写，不难做）
- 数据库层：根据记忆的字段进行分类存储，如短期记忆，长期记忆。
- 遗忘层：引入艾宾浩斯遗忘曲线，数据库中新增访问次数和最后访问时间字段，短期记忆被长时间多次提起，会被拉入长期记忆中。被遗忘的信息将从短期记忆被扔到遗忘表中，这是一个不会被使用的长期记忆，仅在年度记录或用户主动提起的时候，会进入该表检索。

长期记忆将不再拥有权重和次数等信息，直接整合存入rag数据库中长期存放。



# 数据库设计

## ABOUT表

特点：首次登录软件时由用户手动填写，具有最高优先级，任何记忆和ABOUT信息冲突都以ABOUt信息为准。大概为：

- 名字
- 生日
- 时区
- 性别
- 称呼
- 职业
- 学校

等。当用户设置了生日后，每年的生日时，应该主动弹出生日快乐的弹窗，并分析今年内的各种事情。



## Episodes 短期缓存表

特点：将最近的聊天，提取特征写入记忆，如果AI认为是短期记忆则写入该表，生命周期为自记录起的30天，被引用时再延长一定的时间，具体多长时间没想好，得看看艾宾浩斯遗忘曲线和相关论文。

该表的内容被认为时间结束后，会被全部写入遗忘表。

表结构大概如下：

| id   | session_id | content           | important | create_time | fact_time | last_visit_time | access_count | decay_factor | confidence | emotion_code     | embedding                |
| ---- | ---------- | ----------------- | --------- | ----------- | --------- | --------------- | ------------ | ------------ | ---------- | ---------------- | ------------------------ |
| 1    | 1          | {user} 喜欢玩原神 | 0.21      | 2026-07-20  | NULL      | 2026-07-21      | 3            | 0.2          | 0.95       | 1（也就是happy） | "[0.12, -0.34, 0.77...]" |

用户会被转存为{user}，由于短期记忆的以往周期在7-30天，因此不需要记忆太详细的时间，fact time为事实发生的时间，例如：我昨天吃了拉面，我今天打算玩原神。如果出现模糊情况则填写NULL。最后需要做一个embedding来做检索。

需要注意的是，权和遗忘程度并不是相关的，例如：

| content                          | important | dacay_factor |
| -------------------------------- | --------- | ------------ |
| 今天去北京旅游吃了特别好吃的烤鸭 | 0.2       | 1.7          |

很多小事反而能记很久。不过大模型能否辨认这一点，还有待验证。难道说我需要先写个论文做个实验？？？

（我拿deepseek测试了一下，测试结果如下：

| content                          | important（0-1） | dacay_factor（0-1） |
| -------------------------------- | ---------------- | ------------------- |
| 今天去北京旅游吃了特别好吃的烤鸭 | 0.2              | 0.35                |
| 小学时候去马来西亚看了喷泉       | 0.25             | 0.85                |

嗯，看来大模型目前的能力是足够区分这二者的，就不做进一步的实验了。

置信度的计算还没有想好，我的想法是直接让ai分析，如果用户明确提出不可以那么再修改值。



## Semantic Memory 长期记忆表

特点：持久化储存有关用户的信息，这个表会同步变化为rag数据库，也是ai了解用户的核心表。部分信息是由ai直接判断该信息为长期记忆，部分信息是短期记忆被频繁提起（假设 权>x && decay_factor>y && access_count  >z && last_visit_time == today ），那么短期记忆会被写入长期记忆表中（此时直接删除短期记忆条）

由于长期记忆不再被遗忘，因此不会设置与遗忘有关的字段，整个表会简单很多：

| id   | session_id | content              | important | create_time | fact_time  | confidence | emotion |
| ---- | ---------- | -------------------- | --------- | ----------- | ---------- | ---------- | ------- |
| 1    | 1          | {user}认为原神不牛逼 | 0.1       | 2026-07-20  | 2026-07-20 | 0.95       | 1       |

但是，实际上的ai对话中，并不会真正使用这个表，而是把这个表转换为rag数据库，查找rag，这样性能可以得到大幅度的提升。



## People 人物角色表

特点：记录用户的人际关系，当用户吐槽某个人的时候，如果有记忆，ai能够立刻知道这个人大概是什么样的。

人物角色表的遗忘机制比较特殊，认为一个角色在事实描述中已经不再联系了，那么会修改 `is_contact` 字段，当ai发现两个称呼接近的角色时，会优先考虑 `is_contact` 为1的角色。

结构如下：

| id   | character_name | alias_list          | gender | age_desc | identity       | character_tags  | advantage_tags                    | disadvantage_tags             | related_event_list                                           | sentiment_code | confidence | create_time | relation_type | is_contact |
| ---- | -------------- | ------------------- | ------ | -------- | -------------- | --------------- | --------------------------------- | ----------------------------- | ------------------------------------------------------------ | -------------- | ---------- | ----------- | ------------- | ---------- |
| 1    | 小王           | ["王哥","隔壁小王"] | 1      | 20 多岁  | 同部门后端同事 | ["开朗","热心"] | ["乐于帮忙改 bug","奶茶经常请客"] | ["拖延症严重","开会容易走神"] | ["上周接口报错陪我加班调试到 20 点","团建组队一起打游戏","上次帮我领取办公设备"] | [2]            | 0.93       | 2026-06-12  | 1             | 1          |

| 字段名             | 说明                                                         |
| ------------------ | ------------------------------------------------------------ |
| id                 | 自增主键，人物记录唯一标识                                   |
| character_name     | 人物标准主名称，统一基准称呼                                 |
| alias_list         | JSON 数组，存储外号、别称、其他叫法，用于模糊匹配识别同一人  |
| gender             | 性别编码：1 = 男，2 = 女，3 = 未知                           |
| age_desc           | 年龄文字描述，兼容模糊表述（20 多岁 / 中年 / 学生等）        |
| identity           | 人物身份、职业、所属关系定位                                 |
| character_tags     | JSON 数组，存储性格标签，结构化提取人物特质                  |
| advantage_tags     | JSON 数组，单独存储该人物优点，方便 AI 正向评价调取          |
| disadvantage_tags  | JSON 数组，单独存储该人物缺点，吐槽场景优先匹配              |
| related_event_list | JSON 数组，存储和此人发生过的所有相关事件，支持持续追加新增经历 |
| sentiment_code     | JSON 数字数组，记录你对该人物长期情绪倾向：1 平淡、2 好感、3 反感 |
| confidence         | 画像置信度 0~1，代表提取到的人物信息可靠程度                 |
| create_time        | 这条人物档案初次入库时间                                     |
| relation_type      | 关系类型：1 = 长期稳定关系，2 = 短期临时关系。               |
| is_contact         | 是否仍有来往：1 正常联络，0 已断联，断联会加速记忆权重衰减   |



## Reflection AI总结表：

特点：这个表只记录每天发生的事实，不记录用户的观点。例如：我今天滑雪，很开心->{user}在{date}滑雪。

注意的是，这个表只储存发生在 `create_time` 的事情，不会记录“我昨天吃的饭真好吃”（避免事实重复），为了更精确的匹配，这个表会储存的更加细致，例如用户可能会提问：“去年的今天我在做什么呢？”

| id   | fact_content                                        | create_time | memory_weight | confidence | embedding |
| ---- | --------------------------------------------------- | ----------- | ------------- | ---------- | --------- |
| 1    | {user}在{create_time}玩了一整天apex导致忘记了上课。 | 2026-07-21  | 0.6           | 0.95       | ...       |

这有利于ai直接知道最近发生了什么，而不是反复阅读之前的对话窗口。



## Embedding RAG向量数据库：

会储存Semantic Memory 长期记忆表，People 人物角色表与Reflection AI总结表，有利于检索速度优化。
